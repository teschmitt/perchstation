# On-device gate run — 2026-06-18 (perchpi)

A `deploy/RELEASE-CHECKLIST.md` attempt on real hardware. **The gate did not
pass** — it cannot, in this environment (dead camera, no sensor wired, no
reachable mTLS perchpub). This is the deviation log + what was actually
validated, what was fixed, and what remains.

## Environment

| Item | Value | Note |
| --- | --- | --- |
| Device | `perchpi` — Raspberry Pi 4 (BCM2711), 1.8 GiB RAM | **Deviation**: spec/checklist target the Pi Zero 2 W RSS/soak budgets |
| OS | Debian 13 **trixie**, kernel 6.12.75, aarch64 | **Deviation**: spec targets Raspberry Pi OS **Bookworm** |
| Camera | `ov5647` enumerates, capture times out at V4L2 | HW fault: *"Camera frontend has timed out … check that your camera sensor connector is attached securely"* (CSI ribbon/connector) |

| perchpub | `192.168.178.186:8000`, plain **HTTP** (uvicorn / full-stack-fastapi) | mTLS enforced by a front proxy; 443/8443 closed |

## Fixes applied this session (uncommitted, in the working tree)

1. **Camera command is now configurable.** New `[capture].camera_command`
   (default `rpicam-vid`) and `camera_still_command` (default `rpicam-still`),
   wired through `serve.rs` and `enroll.rs` via `.with_binary()`.
   *Was*: `libcamera-vid` / `libcamera-still` hardcoded; absent on trixie
   (only `rpicam-*` ship), so both the recorder and `enroll --qr-source camera`
   would spawn-fail on this OS regardless of the HW fault.
2. **systemd unit device access.** Replaced `PrivateDevices=true` with
   `DeviceAllow=/dev/gpiochip0 rw` + `SupplementaryGroups=gpio video` (plus a
   commented, **unverified** camera-device block).
   *Was*: `PrivateDevices=true` hid `/dev/gpiochip0`, so `GpioMotionSensor::new()`
   failed → `capture.init_failed` → **the entire capture subsystem was dead
   under the production unit on any hardware**. **Verified fixed on-device**
   (`capture.ready`).
3. **systemd unit restart-limit keys.** Moved `StartLimitIntervalSec` /
   `StartLimitBurst` from `[Service]` to `[Unit]`. *Was*: in `[Service]`, where
   systemd ignores them (logged `Unknown key … ignoring`) → restart
   rate-limiting was inactive.

## Validated ✅ (with evidence)

- Binary is correct for the target (ELF aarch64, links only glibc/libm/pthread/dl) and runs.
- Config-bounds validation now fires on `serve` (`clip_duration_secs=0` →
  `out of range: must be in 1..=3600`, exit 70) — confirms the **fresh** binary
  is deployed (the Pi previously ran a Jun-10 binary predating `afbc3da`).
- `service.ready` + `sd_notify(READY=1)` → `active (running)`, `NRestarts=0`.
- **`capture.ready`** — capture loop initialises and opens `/dev/gpiochip0`
  under the hardened unit (validates fix #2).
- Delivery picks up a queued clip and **handles a terminal upload failure
  correctly**: `delivery.upload_terminal status=401` → clip marked
  undeliverable, sidecar moved to `delivered/`, `.mp4` removed, no retry storm.
- Outbound traffic only to the configured perchpub authority.
- Journal hygiene clean (every line valid JSON; no private-key PEM or
  `auth_token` leakage) over the (limited) log volume produced.

## Findings / deviations 🚩

- **A. SC-007 status surface is not wired cross-process.** `perchstation status`
  passes `None` for `CaptureState` (separate process, no shared file/IPC), so
  the capture fields are **always** `never_observed` / `(none)` regardless of
  the running capture loop. **RELEASE-CHECKLIST Step 3.2 cannot pass as
  written.** (`commands/status.rs::run` documents this as intentional-but-incomplete.)
  Capture activity is observable only via the journal (`capture.*` events).
- **B. Enrollment confirm — 502 + client hang (HTTP path).** Against this
  perchpub, the binary's confirm POST gets **HTTP 502** (while `curl` with an
  openssl CSR gets `200` and a signed cert), and reqwest/hyper then **hangs
  ~30 s on the 502 + connection-close** (TCP_USER_TIMEOUT=30000) instead of
  surfacing it, so confirm retries until timeout. May be specific to the
  cleartext/proxy path; the real binary uses mTLS/HTTPS. Needs perchpub-side
  logs and a test against the real mTLS endpoint to judge release relevance.
- **C. perchpub enforces mTLS.** `POST /api/v1/upload/` over HTTP →
  `401 "client certificate required"`. A plaintext smoke cannot complete an
  upload; real interop requires the station client cert over TLS.
- **D. Stale deployed binary (pre-existing, corrected).** The Pi shipped a
  Jun-10 binary predating the config-validation commit; redeployed HEAD.

## Verdict & required follow-ups

The release gate **cannot be promoted** from this run. Before a real gate:

1. Repair the camera (CSI ribbon/connector) — re-test Steps 1 & 3.1.
2. Wire a motion sensor (BCM 17 per config) — re-test 3.1/3.3/3.4.
3. Provide an **mTLS-capable perchpub** reachable from the Pi — re-test Steps 2, 5, 6, 7.
4. **Fix finding A** (wire `status` to the running capture state) — Step 3.2 depends on it.
5. Investigate **finding B** (confirm 502 + reqwest hang) against the real mTLS endpoint.
6. Commit the three fixes above (prerequisites for any capture-side testing on trixie).

> The HTTP smoke required a throwaway, **insecure** binary (https-only disabled
> in both clients, CA-pin disabled) plus fabricated credentials. It is not a
> release artifact and was removed during tear-down; the RC binary
> (HEAD + the three fixes) is what remains installed.
